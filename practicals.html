Prac 1 - HTML (non-CMS) Web Design

Task: Your task in the first prac is to make a very simple website for your startup business for assignment 1 and publish it online (submit the URL). The site should have at least 3 pages (or sections for a single-page design) and contain enough content that a reasonable person would understand your startup business; including text and images.

Your assignments are to be done with a CMS, but this prac asks you to create a static website.
You are welcome to use Bootstrap or any other non-CMS technology, including templates/examples you find online, BUT you may NOT use a site builder like Wix, SquareSpace, Weebly... You need to write HTML yourself as that's the learning outcome here.

Consider the content and design that your site would need to be able to achieve its goal... and consider what that goal is!

Prac 4 - Local Development + Deployment

Task: get a local development environment setup and be able to deploy your local sites to "production". Note that you don't have to have this totally figured out today; it will take time to experiment with different options and adjust your system as you learn and try new things. If you have already got a local development + deployment workflow going, then continue to work on your sites according to the assignment requirements (e.g. have you got plugins/extensions installed?).

Prac 5 - WordPress Child Theme

Task: using either your first assignment or another WordPress site, create a child theme with at least 6 visible customisations.

If you're using VVV and the topdown dashboard, you can use it to start your new child theme.
Otherwise, follow the developer docs to create a folder inside themes with the recommended naming convention, copy or create a new style.css file (and remember to refer to the parent theme in the comment, e.g. if the parent theme were "twentysixteen", Template: twentysixteen), and create functions.php to enqueue the style file(s), as explained in class (see notes).

Use your browser's inspector to make live changes, then copy these to your CSS file.

Prac 6 - Group Setup

Task: complete setting up all of the teamwork tools, make at least one GitHub commit (see a message in your integrated/linked Slack group), and complete your group agreement in your project board.

Please meet in your teams to talk about your assignment 2 project team and to start getting organised:

  • You will have been added to a private group on Slack (check this now)
  • (everyone) Login to GitHub, then create or join your group's GitHub repo using the GitHub Classroom link provided in the assignment details.
    If you are first to create the GitHub repo, then when asked for your repo name, it should be the same as your Slack group name so that it's easy to find (e.g. cp3402-2018-team01).
  • Option: If you would prefer to use GitHub Projects instead of Trello, that is fine. GitHub Projects is more suitable if you want to focus on tasks more in terms of code and issues.
    (one person) copy the Trello template board, make sure it is public and add all of the other team members.
    You do not need to use your Trello board in the way that this template is set out - it's just an example. How you do things is up to you, but the group agreement needs to stay on your board. Everything else is your choice.

Group Project Work Pracs

Part of the marks for each of the remaining practicals are for effective participation in your stand-up + work done on your project.
You should usually be able to demonstrate work by GitHub commits, and we should be able to see a history of completed work in your project board.
Note that some remaining pracs have extra requirements for individual work in addition to the group work component.

In these later practicals, you will work in your project teams to develop the project assignment.

Think of each week as a sprint.
Each of these practicals will start with a stand-up meeting for each group.
Here is a short video about stand-ups to help everyone understand them.

Make sure you actually stand for your meeting! This IS what many industry teams do every single day, so this is preparing you for industry.
These meetings are NOT for you to present to the lecturer – it is for you to discuss your work; and the lecturer/tutor will just witness the meetings (and possibly add prompts if needed).

Here’s how each stand-up should go. Each person tells the rest of their team:

  • What they worked on last week (since the last meeting)
  • Any road-blocks or things they’re stuck on (and might need help with)
  • What they plan to achieve in the coming week (before the next meeting)

Be honest about your progress, and help each other. Discuss who is doing what, and then use your project board to assign tasks (use "members").

After your short (no more than 5 minutes) meeting, get to work, developing your project site, including working out and using your development and deployment workflow.

Make sure you continue to research modern development workflows and adjust your team's workflow as you learn, and based on your experiences. You should be working with multiple local development environments (one per member), a staging and a production environment.

Use GitHub to keep track of and version control your code outputs. You should probably set your repo to track your wp-content folder. Use a .gitignore and/or remove files that you don't need to track (like other themes).

Prac 7 - PHP (& Group Project Work)

Task: write and publish PHP code to demonstrate the features below (the content of the page/site is up to you but must be original, not from some PHP tutorial).

  • echoing different kinds of HTML
  • selection - if/else statements
  • loops - for, while, foreach (use an array)
  • functions with parameters
  • include - make two pages that include a common header and/or footer

You may like to fork The Advice Shop site, get it running, then update it by making a proper login system and other nice enhancements.

Prac 9 - Sass (& Group Project Work)

Task: demonstrate your ability to use Sass and a suitable workflow (preferably automated) for creating CSS, including the features below in a simple new website (not WordPress). The content is up to you but must be original, not from some Sass tutorial.

  • variables
  • nesting
  • mixins
  • extend/inheritance

Note: To use Sass on the JCU Lab Macs:

  • As the installation instructions say, you can install Sass by downloading the package for your operating system from GitHub and adding it to your PATH.
  • Test it like (example) sass input.scss output.css
    You can then watch for changes, like sass --watch styles:styles (where styles is both the input and output directory)
  • Optional: If you like you can install a task runner like Grunt or Gulp in the Linux VM, then setup a Gruntfile (e.g. like this one) and run grunt/gulp in your VM terminal.
  • You can then use whatever editor you want to edit your Sass code (e.g. PHPStorm on the Mac) and Grunt or the watcher will auto-update your CSS code when it detects changes.
  • Alternatively, you could install Sass in a Linux VM like the one that has been customised to suit the labs: https://github.com/lindsaymarkward/WPDistillery

Penultimate Prac - Presentation Rehearsals

Task: stand-up meeting + group rehearsal of presentation

Don't worry if your presentation is not "finished" at this stage. You can start rehearsing with an unfinished plan or slide deck, and you will more readily see where your gaps are. The more you rehearse, the better you'll be.

Aim to run through your presentation 3 times. After each time, discuss the areas that need improvement, then improve those and rehearse again. Use the assignment document for details of requirements and expectations. As you evaluate your group's practice presentations, you can "mark" them with the supplied rubric - to help identify areas to work on.

Remember to plan and practise a realistic demonstration.

Vagrant

Vagrant is a fabulous tool for configuring and running Virtual Environments on your local computer. You will need VirtualBox

Here are some things to get started with (install):

  • First, here's a good overview of Vagrant, VVV and VV
  • 3 popular Vagrant configurations for Wordpress are:
    • WPDistillery: https://wpdistillery.org/ (uses ScotchBox)
    • Lindsay's version of WPDistillery customised to use http://localhost:8080 for use in JCU's Mac labs: https://github.com/lindsaymarkward/WPDistillery
    • VCCW: http://vccw.cc/
    • VVV: https://github.com/Varying-Vagrant-Vagrants/VVV and VV Site Wizard: https://github.com/bradp/vv, Add-on Dashboard for VVV: https://github.com/topdown/VVV-Dashboard
  • Here's Lindsay's Vagrant project for running WordPress on AWS (see the README for more details)
  • WordMove: a handy tool for copying (deploying) from your Vagrant localhost environment to another (staging/production) site: https://github.com/welaika/wordmove
    You install this IN Vagrant - after vagrant ssh
    Note: if you are using VVV with ruby 2.4, you need to run rvm group add rvm "$USER" then log out and back in again before installing wordmove.
    If you get an error something like "sql_adapter.rb:44:in `gsub!': invalid byte sequence in US-ASCII", edit your .profile file as described in the Wordmove wiki
  • Mac GUI for managing Vagrant boxes: http://vagrantmanager.com
  • And a popular Vagrant box for a more generic Web server (LAMP stack; no CMS installed): https://github.com/scotch-io/scotch-box
