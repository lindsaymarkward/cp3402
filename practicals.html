Prac 5 - WordPress Child Theme

Task: Using either your first assignment or another WordPress site, create a child theme with at least 6 customisations.

If you're using VVV and the topdown dashboard, you can use it to start your new child theme.
Otherwise, follow the developer docs to create a folder inside themes with the recommended naming convention, copy or create a new style.css file (and remember to refer to the parent theme in the comment, e.g. if the parent theme were "twentysixteen", Template: twentysixteen), and create functions.php to enqueue the style file(s).

Use your browser's inspector to make live changes, then copy these to your CSS file.

Prac 6 - Group Setup

Task: complete setting up all of the teamwork tools, make at least one GitHub commit (see a message in your Slack group), and complete your group agreement in your project board.

Please meet in your teams to talk about your assignment 2 project team and to start getting organised:

  • You will have been added to a private group on Slack (check this now)
  • (everyone) Login to GitHub, then create or join your group's GitHub repo using the provided GitHub Classroom link.
    If you are first to create the GitHub repo, then when asked for your repo name, it should be the same as your Slack group name so that it's easy to find (e.g. cp3402-2018-team01).
  • Option: If you would prefer to use GitHub Projects instead of Trello, then that is fine.
    (one person) copy the Trello template board, make sure it is public and add all of the other team members.
    You do not need to use your Trello board in the way that this template is set out - it's just an example. How you do things is up to you, but the group agreement needs to stay on your board. Everything else is your choice.
  • (one person) Integrate both your GitHub repo and Trello board with your Slack group (use "integrations" in Slack). More details (inc. screenshots) for integrating GitHub and Trello with your Slack channel are here.
    When you add an integration (GitHub, Trello Alerts), it might warn you that we've reached the integration limit. This is not a problem because we're just adding more instances of existing integrations. Ignore the warning (but don't try and add an integration for a different app).

    This is how you decide on your shared team expectations, and is an important step. Don't skip through it, but think about it carefully.
  • (everyone) Start doing some project planning, including adding team members to tasks on your Trello board...
    One example of a suitable task for each member to do this week would be to investigate and test different hosting platforms.

Group Project Work Pracs

Marks for the remaining practicals are for effective participation in your stand-up + work done on your project.
You should usually be able to demonstrate work by GitHub commits, and we should be able to see a history of completed work in your project board.
Note that some remaining pracs have extra requirements for individual work in addition to the group work component.

In these later practicals, you will work in your project teams to develop the project assignment.

Think of each week as a sprint.
Each of these practicals will start with a stand-up meeting for each group.
Here is a short video about stand-ups to help everyone understand them.

Make sure you actually stand for your meeting! This IS what many industry teams do every single day, so this is preparing you for industry.
These meetings are NOT for you to present to the lecturer – it is for you to discuss your work; and the lecturer/tutor will just witness the meetings (and possibly add prompts if needed).

Here’s how each stand-up should go. Each person tells the rest of their team:

  • What they worked on last week (since the last meeting)
  • Any road-blocks or things they’re stuck on (and might need help with)
  • What they plan to achieve in the coming week (before the next meeting)

Be honest about your progress, and help each other. Discuss who is doing what, and then use your project board to assign tasks (use "members").

After your short (no more than 5 minutes) meeting, get to work, developing your project site, including working out and using your development and deployment workflow.

Make sure you continue to research modern development workflows and adjust your team's workflow as you learn, and based on your experiences. You should be working with multiple local development environments (one per member), a staging and a production environment.

Use GitHub to keep track of and version control your code outputs. You should probably set your repo to track your wp-content folder. Use a .gitignore and/or remove files that you don't need to track (like other themes).

Penultimate Prac - Presentation Rehearsals

Task: stand-up meeting + group rehearsal of presentation

Don't worry if your presentation is not "finished" at this stage. You can start rehearsing with an unfinished plan or slide deck, and you will more readily see where your gaps are. The more you rehearse, the better you'll be.

Aim to run through your presentation 3 times. After each time, discuss the areas that need improvement, then improve those and rehearse again. Use the assignment document for details of requirements and expectations. As you evaluate your group's practice presentations, you can "mark" them with the supplied rubric - to help identify areas to work on.

Remember to plan and practise a realistic demonstration.

Vagrant

Vagrant is a fabulous tool for configuring and running Virtual Environments on your local computer. You will need VirtualBox

Here are some things to get started with (install):

  • First, here's a good overview of Vagrant, VVV and VV
  • VVV for WordPress: https://github.com/Varying-Vagrant-Vagrants/VVV
  • VV Site Wizard: https://github.com/bradp/vv
  • Add-on Dashboard: https://github.com/topdown/VVV-Dashboard
  • WordMove: a handy tool for copying (deploying) from your Vagrant localhost environment to another (staging/production) site: https://github.com/welaika/wordmove
    You install this IN Vagrant - after vagrant ssh
    Note: if you are using VVV (with ruby 2.4), you need to run rvm group add rvm "$USER" then log out and back in again before installing wordmove.
    If you get an error something like "sql_adapter.rb:44:in `gsub!': invalid byte sequence in US-ASCII", edit your .profile file as described in the Wordmove wiki
  • Mac GUI for managing Vagrant boxes: http://vagrantmanager.com
  • And a popular Vagrant box for a more generic Web server (LAMP stack; no CMS installed): https://github.com/scotch-io/scotch-box
  • Lindsay's version of WPDistillery that sets up WordPress on a scotch-box VM and redirects http://localhost:8080 to the VM for use in our labs: https://github.com/lindsaymarkward/WPDistillery
  • Here's Lindsay's Vagrant project for running WordPress on AWS (see the README for more details)
